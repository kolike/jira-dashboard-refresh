# Jira Fast Watcher Pro — технический разбор проекта

## 1) Текущая структура

Проект сейчас собран как **минимальное Chrome Extension MV3** без сборщика и без внешних зависимостей:

- `manifest.json` — конфигурация расширения (permissions, host permissions, background service worker, popup, content script).
- `background.js` — обработка уведомлений и переход по клику на тикет.
- `content.js` — основная логика мониторинга Jira-виджетов во фреймах, детект новых задач, режим выбора фреймов.
- `popup.html` + `popup.js` — UI управления (вкл/выкл, выбор фреймов, сброс, интервал, tutorial).
- Иконки (`icon*.png`, `icon128_red.png`).
- `jfw3.5_090426.zip` — артефакт/архив релиза (в рантайме не используется).

Архитектурно данные и управление держатся на `chrome.storage.local` + обмен сообщениями между `popup` ↔ `content` ↔ `background`.

## 2) Что уже реализовано

### Базовый каркас расширения
- Manifest V3, service worker, popup и content script подключены корректно.
- Ограничение по хосту Jira задано через `host_permissions` и `matches`.

### Мониторинг новых задач
- В `content.js` реализован цикл обновления фреймов по интервалу.
- Есть первичная инициализация `knownIssues`, чтобы не спамить старыми тикетами при первом запуске.
- Считываются ключ тикета и summary из строк таблиц внутри iframe, формируется событие `NEW_ISSUE`.

### Типизация уведомлений
- Присутствует классификация widget type: `priority` / `transient` / `unknown` (по заголовку виджета и fallback по тексту).
- В `background.js` уведомления отрисовываются с разными иконками и поведенческими настройками:
  - priority — красная иконка, `requireInteraction: true`.
  - transient — синяя иконка, автоочистка таймером.

### UX в popup
- Переключатель состояния мониторинга.
- Выбор интервала обновления.
- Интерактивный режим «прицел» для выбора отслеживаемых гаджетов.
- Сброс выбранных селекторов.
- Счетчик выбранных блоков.
- Встроенный tutorial-оверлей.

## 3) Сломанные / недоделанные части

Ниже проблемы, которые либо уже приводят к сбоям, либо блокируют стабильный MVP.

### Критичные
1. **Мониторинг по умолчанию не стартует после установки.**
   - `onInstalled` выставляет `enabled: true`, но в `content.js` запуск делается только если `data.enabled` уже true на момент чтения storage; на существующих вкладках после установки/обновления это состояние может не синхронизироваться ожидаемо.
   - Нет надежного механизма «гарантированно включить start() при первом заходе на страницу после install/update».

2. **Неполный lifecycle для picker-панели.**
   - При `done` выполняется `location.reload()`, но перед этим не снимаются слушатели и не очищаются подсветки в явном виде (расчет на перезагрузку страницы).
   - В реальных условиях при ошибке reload/на SPA-переходах это может оставлять мусорный UI/handler.

3. **Сильная зависимость от структуры Jira dashboard + iframe same-origin.**
   - Логика жестко завязана на `div.gadget...` и таблицы в iframe.
   - Нет диагностик, если iframe недоступен (CSP/измененный DOM/пустой iframe).

### Высокий приоритет
4. **Отсутствует обработка удаления/закрытия задач из knownIssues.**
   - `knownIssues` только растет в рамках сессии, возможны ложные пропуски при изменениях данных и долгой работе вкладки.

5. **Нет валидации/нормализации custom селекторов.**
   - При изменении DOM ID/перестройке гаджетов сохраненные селекторы silently ломаются.

6. **Невнятная обработка `unknown` типа уведомления.**
   - В background unknown не автозакрывается (по условию `msg.notificationType !== 'unknown'`), что может приводить к «залипанию» уведомлений для неопознанных виджетов.

7. **Логика tutorial в popup дублируется.**
   - В HTML есть статический `#tutorialOverlay`, а в JS создается отдельный динамический overlay `#tutOverlay`.
   - Это технический долг, усложняющий поддержку UI.

### Средний приоритет
8. **Нет логирования и режима диагностики.**
   - Сейчас ошибки почти везде «глотаются» `catch(e) {}` без следа.

9. **Нет минимального smoke/e2e и даже script-based self-check.**
   - Регрессии по DOM Jira и messaging будут находиться только вручную.

10. **Нет документации по установке/использованию/ограничениям.**
   - Для MVP не хватает `README` с понятным флоу и known limitations.

## 4) Пошаговый план до рабочего MVP

Принцип: **не переписывать архитектуру**, только довести текущую реализацию до стабильного и поддерживаемого состояния.

### Этап 0 — Зафиксировать базу (0.5 дня)
1. Добавить `README.md` с инструкцией загрузки unpacked extension, сценариями использования и ограничениями (same-origin iframe, поддерживаемые страницы Jira).
2. Добавить раздел troubleshooting (что делать, если не приходят уведомления).

### Этап 1 — Стабилизация ядра (1 день)
3. Починить инициализацию `enabled/start` для надежного старта на активной Jira-странице.
4. Добавить защищенный lifecycle picker (снятие listeners + очистка outline в любом выходе).
5. Добавить минимальный debug-флаг в storage и контролируемый `console.debug` вместо silent-catch.

### Этап 2 — Надежность детекции тикетов (1–1.5 дня)
6. Добавить валидацию `customSelectors` и мягкий fallback при битых селекторах.
7. Сделать более устойчивый парсинг rows/links (явно проверять key pattern типа `ABC-123`).
8. Пересмотреть стратегию `knownIssues` (например, windowed cache + periodic resync), чтобы не терять новые тикеты на длинных сессиях.

### Этап 3 — UX полировка без редизайна (0.5–1 день)
9. Убрать дубли tutorial (оставить один источник правды).
10. Явно отобразить в popup состояние: «фреймы не выбраны / выбранные селекторы не найдены на странице».
11. Привести поведение `unknown` уведомлений к ожидаемому (или автозакрытие, или отдельная пометка в UI).

### Этап 4 — Минимальная проверка качества (0.5 дня)
12. Добавить npm-free check-скрипт (lint-like) или хотя бы документированный smoke checklist:
    - включение/выключение,
    - выбор/сброс фреймов,
    - уведомления priority/transient,
    - переход по клику на тикет.
13. Пройти ручной прогон на 2–3 реальных дашбордах Jira.

## 5) Что НЕ нужно делать для MVP (чтобы не переписывать лишнее)

- Не внедрять сборщик (webpack/vite) только ради структуры.
- Не переходить на тяжелую state-management архитектуру.
- Не делать полную абстракцию под «любую Jira» до стабилизации текущего домена.
- Не трогать визуальный дизайн popup, кроме устранения дубликатов и статусов ошибок.

---

## Итог

Сейчас проект уже близок к рабочему utilitarian MVP: основной pipeline (scan → detect → notify) существует. Главный риск — хрупкость DOM-интеграции и отсутствие диагностик. Если выполнить план выше, можно получить стабильный MVP без переписывания расширения «с нуля».
